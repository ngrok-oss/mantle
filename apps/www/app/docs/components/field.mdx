# Field

`Field` is a compound layout primitive for building a single form field. It groups the existing mantle [`Label`](/components/label) with a control (Input, Select, Checkbox, etc.) and adds slots for helper / hint text (`Field.Description`) and validation errors (`Field.Error`, `Field.ErrorList`). Stack multiple fields together with `Field.Group`, or ‚Äî for radios and related checkboxes specifically ‚Äî wrap them in a `Field.Set` + `Field.Legend` for a properly-named semantic group.

`Field` does not invent a new label component ‚Äî pair `Field.Item` with the existing `<Label>`, which already handles `htmlFor`, click-to-focus, and the auto `font-medium` typography.

`Field.Item` automatically gives descendant `Field.Description` and non-empty `Field.ErrorList` parts stable IDs, then merges those IDs onto known form controls via `aria-describedby` and `aria-errormessage`. It does not change the control's `id` or `name`, so `id={field.name}` stays the right TanStack Form pattern.

<Example className="w-full">
	<Field.Item className="max-w-80">
		<Label htmlFor="field-username">Username</Label>
		<Input id="field-username" name="username" placeholder="ada-lovelace" />
		<Field.Description>Pick something memorable ‚Äî you can change it later.</Field.Description>
	</Field.Item>
</Example>

```tsx
import { Field } from "@ngrok/mantle/field";
import { Input } from "@ngrok/mantle/input";
import { Label } from "@ngrok/mantle/label";

<Field.Item className="max-w-80">
	<Label htmlFor="username">Username</Label>
	<Input id="username" name="username" placeholder="ada-lovelace" />
	<Field.Description>Pick something memorable ‚Äî you can change it later.</Field.Description>
</Field.Item>;
```

## Error messages

Wrap one or more `Field.Error` children in a `Field.ErrorList` below the control when validation fails. The list renders a semantic `<ul>` with `<li>` items so screen readers announce the errors as a list, and a single error is just a list of one. Set `validation="error"` on the control so the visual states stay in sync; `Field.Item` handles the `aria-errormessage` association and marks known controls invalid when an error list is present.

When a `Field.Description` follows directly after a `Field.ErrorList`, `Field.Description` automatically collapses the parent's `gap-1.5` with a matching negative top margin so the errors + helper read as one tight block ‚Äî matching the figma. Pass any margin utility on `Field.Description` (e.g. `className="mt-1"`, `className="mt-0"`) to override.

<Example className="w-full">
	<Field.Item className="max-w-80">
		<Label htmlFor="field-email-error">Email</Label>
		<Input
			id="field-email-error"
			name="email"
			type="email"
			defaultValue="not-an-email"
			validation="error"
		/>
		<Field.ErrorList>
			<Field.Error>Enter a valid email address.</Field.Error>
		</Field.ErrorList>
		<Field.Description>We'll never share your email.</Field.Description>
	</Field.Item>
</Example>

```tsx
<Field.Item>
	<Label htmlFor="email">Email</Label>

		<Field.Error>Enter a valid email address.</Field.Error>
```

## Multiple errors

Render multiple `Field.Error` siblings inside `Field.ErrorList` when a validator produces more than one message at once (e.g. `minLength` + `pattern` + custom rule). Items stack tightly so the whole error block reads as one unit.

## Optional fields

Use `Field.Optional` inside a `<Label>` to mark a field as optional. It renders a muted "(Optional)" suffix at `text-sm` regular weight so it reads as secondary metadata without competing with the bolder label text. Defaults to the literal "(Optional)" ‚Äî pass children to translate or replace.

<Example className="w-full">
	<Field.Item className="max-w-80">
		<Label htmlFor="field-nickname" className="flex items-baseline gap-1">
			Nickname <Field.Optional />
		</Label>
		<Input id="field-nickname" name="nickname" placeholder="ada" />
		<Field.Description>Visible on your public profile.</Field.Description>
	</Field.Item>
</Example>

```tsx
<Field.Item>
	<Label htmlFor="nickname" className="flex items-baseline gap-1">
		Nickname <Field.Optional />
	</Label>
	<Input id="nickname" name="nickname" placeholder="ada" />
```

Place `Field.Optional` inside the `<Label>` so screen readers announce it as part of the accessible name (e.g. "Nickname, Optional, edit text"). The label's `flex items-baseline gap-1` keeps the suffix on the same baseline as the bolder label text.

## Help icon next to the label

When the label needs a sibling affordance that can't live inside the `<Label>` itself ‚Äî most commonly an interactive help-icon button ‚Äî wrap the label and its decorations in `Field.LabelRow`. It's a horizontal flex container with `items-center` and `gap-1` so the label, optional suffix, and help icon all sit on a shared center line. (Center alignment, not baseline ‚Äî SVG icon buttons have no text baseline, so `items-baseline` would push them above the label text.)

Putting an interactive button inside `<label>` is a footgun: clicks on a `<label>` forward focus to the associated control, so a help button nested in a `<Label>` would steal that behavior. `Field.LabelRow` is the way out.

Reach for the bundled `Field.Help` / `Field.HelpTrigger` / `Field.HelpContent` rather than wiring `Popover` + `IconButton` + an icon by hand ‚Äî it's the same composition, three thin wrappers, with sensible defaults (ghost `IconButton`, `xs` size, default Phosphor `QuestionIcon`, `text-body` color). [`Popover`](/components/popover) under the hood ‚Äî not [`Tooltip`](/components/tooltip) ‚Äî so the affordance is reachable on touch devices.

<Example className="w-full">
	<Field.Item className="max-w-80">
		<Field.LabelRow>
			<Label htmlFor="field-api-key" className="flex items-baseline gap-1">
				API key <Field.Optional />
			</Label>
			<Field.Help>
				<Field.HelpTrigger label="What is an API key?" />
				<Field.HelpContent>Copy this from the ngrok dashboard.</Field.HelpContent>
			</Field.Help>
		</Field.LabelRow>
		<Input id="field-api-key" name="apiKey" placeholder="2A1b‚Ä¶" />
		<Field.Description>You can find this in the ngrok dashboard.</Field.Description>
	</Field.Item>
</Example>

## Radio and checkbox groups

`Field.Set` renders a real `<fieldset>` ‚Äî paired with `Field.Legend`, it gives a related set of controls a single accessible name (so a screen reader announces "Notification frequency, group, Daily"). Reach for it specifically when the grouping is itself the question the user is answering: most commonly a [`RadioGroup`](/components/radio-group) (one question, many possible answers), or a set of related checkboxes (preference toggles, permission flags, opt-ins).

**When _not_ to use it.** Stacking unrelated fields under a shared header is _not_ a semantic group ‚Äî `Field.Group` (above) is the right call. A `<fieldset>` around plain inputs reads as noise to assistive tech and looks visually heavy. If you find yourself reaching for `Field.Set` to add a section title to a form, render a regular heading instead.

Add a `Field.Description` directly below the `Field.Legend` to clarify the question ‚Äî the legend itself is short by design, so a one-liner below it carries the context. (Chromium renders the legend with no flex gap to its next sibling, so the description naturally hugs the legend without extra styling.)

## With TanStack Form

`Field` is designed to slot directly into [TanStack Form](https://tanstack.com/form). `field.name` is a stable, unique string for each field ‚Äî pass it as both `htmlFor` on the `Label` and `id` on the control to wire the label association without a separate `useId()` call. `Field.Item` leaves that `id` alone while auto-wiring descendant descriptions and errors. Map `field.state.meta.errors` (e.g. `ZodIssue[]`) inside `Field.ErrorList` to render one `Field.Error` per message; the list renders nothing when the array is empty so it's safe to leave mounted unconditionally:

```tsx
import { Field } from "@ngrok/mantle/field";
import { Input } from "@ngrok/mantle/input";
import { Label } from "@ngrok/mantle/label";
import { Button } from "@ngrok/mantle/button";
import { useForm } from "@tanstack/react-form";
import { z } from "zod";

const schema = z.object({
	email: z.email("Enter a valid email."),
	password: z.string().min(12, "Must be at least 12 characters."),
});

export function AccountForm() {
	const form = useForm({
		defaultValues: { email: "", password: "" },
		validators: { onChange: schema },
		onSubmit: ({ value }) => {
			console.log(value);
		},
	});

	return (
		<form
			className="flex flex-col gap-4"
			onSubmit={(event) => {
				event.preventDefault();
				event.stopPropagation();
				void form.handleSubmit();
			}}
		>
			<Field.Group>
				<form.Field name="email">
					{(field) => (
						<Field.Item>
							<Label htmlFor={field.name}>Email</Label>
							<Input
								id={field.name}
								name={field.name}
								type="email"
								value={field.state.value}
								onBlur={field.handleBlur}
								onChange={(event) => field.handleChange(event.target.value)}
								validation={field.state.meta.errors.length > 0 ? "error" : false}
							/>
							<Field.ErrorList>
								{field.state.meta.errors.map((error, index) => (
									<Field.Error key={index}>{error?.message}</Field.Error>
								))}
							</Field.ErrorList>
							<Field.Description>We'll never share your email.</Field.Description>
						</Field.Item>
					)}
				</form.Field>
				<form.Field name="password">
					{(field) => (
						<Field.Item>
							<Label htmlFor={field.name}>Password</Label>
							<Input
								id={field.name}
								name={field.name}
								type="password"
								value={field.state.value}
								onBlur={field.handleBlur}
								onChange={(event) => field.handleChange(event.target.value)}
								validation={field.state.meta.errors.length > 0 ? "error" : false}
							/>
							<Field.ErrorList>
								{field.state.meta.errors.map((error, index) => (
									<Field.Error key={index}>{error?.message}</Field.Error>
								))}
							</Field.ErrorList>
							<Field.Description>At least 12 characters.</Field.Description>
						</Field.Item>
					)}
				</form.Field>
			</Field.Group>
			<Button type="submit" appearance="filled">
				Submit
			</Button>
		</form>
	);
}
```

Most forms only need a `Field.Group` of `Field.Item`s ‚Äî that's the default tree:


## Polymorphism

Layout and supplemental `Field.*` parts accept an `asChild` prop. When `true`, the part renders its single child instead of its default element, forwarding all class names, `data-*` attributes, and the ref onto that child. `Field.Set`, `Field.Legend`, and `Field.Error` always render `<fieldset>`, `<legend>`, and `<li>` respectively, because those elements are the semantics those parts exist to provide.

```tsx
<Field.ErrorList asChild>
	<ol>
		<Field.Error>Fix this first.</Field.Error>
		<Field.Error>Then fix this.</Field.Error>
	</ol>
</Field.ErrorList>
```

## API Reference

### Field.Item

A single form field. Renders a plain `<div>` with `flex flex-col gap-1.5` so a Label, control, and helper / error siblings stack tightly together. No implicit ARIA role ‚Äî the `<label htmlFor>` ‚Üî control association already provides the right semantics for a single field. Descendant `Field.Description` IDs are merged into known controls' `aria-describedby`; non-empty `Field.ErrorList` IDs are merged into `aria-errormessage` and mark known controls invalid when they have not provided their own `aria-invalid`.

**When to use:** for every individual field ‚Äî text inputs, selects, single checkboxes, switches, password inputs, etc.

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Group

Layout container that stacks multiple `Field.Item`s vertically with `gap-4`. Renders a plain `<div>` ‚Äî pure layout, no semantics.

**When to use:** any time a form has more than one field. This is the default way to compose multiple `Field.Item`s. Reach for `Field.Set` + `Field.Legend` instead only when the grouping itself carries semantic weight (radios, related checkboxes).

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Set

Renders a semantic `<fieldset>` with default browser styling reset (no border, no padding, `min-width: 0`). Pair with `Field.Legend` to give the surrounding controls a single accessible name.

**When to use:** specifically for radios (one question, many possible answers) or sets of related checkboxes (preference toggles, permission flags). _Skip it_ for stacking unrelated fields ‚Äî `Field.Group` is the right call there. A `<fieldset>` around plain inputs reads as noise to assistive tech.

All props from [fieldset](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/fieldset#attributes).

### Field.Legend

Caption for a `Field.Set`. Renders a `<legend>` styled to match the mantle `Label` typography. Always render a `Field.Legend` inside a `Field.Set` ‚Äî it's what gives the surrounding fieldset its accessible name.

All props from [legend](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/legend#attributes).

### Field.LabelRow

Horizontal layout container for the label area of a field ‚Äî a flex row with `items-center` and `gap-1`. Use when the label needs a sibling affordance that can't live inside the `<Label>` itself, like a help-icon `Field.HelpTrigger`. Center-aligned (not baseline) so SVG icon buttons sit visually centered next to the label text.

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Help

`Popover.Root` wrapper for the help-affordance pattern. Composes with `Field.HelpTrigger` and `Field.HelpContent`. Forwards all `Popover.Root` props (`open`, `defaultOpen`, `onOpenChange`, `modal`, etc.).

### Field.HelpTrigger

`Popover.Trigger` wired to a ghost-appearance `IconButton` with a default Phosphor `QuestionIcon`. Pre-styled with `text-body` so the icon reads as subtle metadata at rest; `IconButton`'s ghost `hover:text-strong` brightens it on interaction.

Accepts every `IconButton` prop ‚Äî they all become optional with sensible defaults:

| Prop          | Type                       | Default            | Description                                                          |
| ------------- | -------------------------- | ------------------ | -------------------------------------------------------------------- |
| `icon?`       | `ReactNode`                | `<QuestionIcon />` | Icon rendered inside the trigger button. Override to swap the glyph. |
| `label?`      | `string`                   | `"Show help"`      | Visually-hidden accessible label announced by screen readers.        |
| `appearance?` | `"ghost"` \| `"outlined"`  | `"ghost"`          | `IconButton` appearance variant.                                     |
| `size?`       | `"xs"` \| `"sm"` \| `"md"` | `"xs"`             | `IconButton` size.                                                   |
| `className?`  | `string`                   | ‚Äî                  | Merged onto the underlying `IconButton`.                             |

### Field.HelpContent

Body of a `Field.Help` popover. Forwards every `Popover.Content` prop ‚Äî see the [Popover docs](/components/popover#popovercontent) for the full surface (`side`, `align`, `preferredWidth`, etc.).

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Optional

Inline "(Optional)" suffix to mark a field as optional. Renders a `<span>` in `text-muted` at `text-sm` / `font-normal`. Default content is the literal string "(Optional)" ‚Äî pass children to translate or replace.

All props from [span](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/span#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Description

Helper / hint text. Renders a `<p>` in the muted body color. Works in two positions:

1. **Inside `Field.Item`**, below the control ‚Äî clarifies expected format or constraints for that single field.
2. **Inside `Field.Set`, directly below the `Field.Legend`** ‚Äî clarifies the question the legend names (most useful with radios or checkbox groups).

When rendered inside `Field.Item`, `Field.Description` receives a generated ID unless you provide one, and that ID is merged into the control's `aria-describedby`. When `Field.Description` is rendered immediately after a `Field.ErrorList`, it automatically collapses the parent's `gap-1.5` via a `-mt-1.5` so errors + helper read as a single tight block. The rule's specificity is flattened to `(0,1,0)` so any margin utility you pass on `Field.Description` (e.g. `mt-1`, `mt-0`) overrides cleanly. After a `Field.Legend` no negative margin is needed ‚Äî Chromium renders the legend with no flex gap to its next sibling, so the description naturally hugs the legend.

All props from [p](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/p#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Field.Error

A single error message for a field. Renders an `<li>` in `text-danger-600`, so it must be nested inside a `Field.ErrorList`. A single error is just a list of one.

All props from [li](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/li#attributes).

### Field.ErrorList

Wraps one or more `Field.Error` children in a semantic `<ul>` with `role="list"` so a list of validation errors is announced as a list by screen readers, including Safari/VoiceOver combinations that drop list semantics when list styling is removed. Renders nothing when given no children, so it can be left mounted while a validator produces a (possibly empty) error array. When rendered inside `Field.Item`, a non-empty list receives a generated ID unless you provide one, and that ID is merged into the control's `aria-errormessage`. Strips the default browser `<ul>` styling (no bullets, no padding, no margin) and stacks items as a flex column with no gap so consecutive errors read as a single tight block. Pass `asChild` with an `<ol>` if the list is meaningfully ordered.

All props from [ul](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/ul#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |
