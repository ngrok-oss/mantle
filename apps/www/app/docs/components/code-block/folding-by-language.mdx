# Code Block Folding by Language

Mantle picks one of three folding strategies based on the language:

- **Bracket-paired** — multi-line `{ … }` and `[ … ]` ranges fold via TextMate-scope-aware token walking. Brackets inside strings, comments, or regular expressions are ignored. Used for `javascript`, `typescript`, `jsx`, `tsx`, `json`, `css`, `go`, `java`, `csharp`, `rust`, `ruby`.
- **Indentation-based** — blocks are detected by leading whitespace; the opener line stays visible and child lines collapse. Used for `python`, `yaml`.
- **Tag-based** — multi-line elements fold by matching `<tag>` / `</tag>` pairs. Self-closing tags and HTML void elements don't open folds. Used for `html`, `xml`.

`bash`, `shell`, and plain text don't fold today — shell languages use keyword pairs (`if … fi`, `do … done`) we don't yet model, and plain text has nothing to fold.

## Caveats

- **HTML / XML** — multi-line opening tags (where `<` and `>` sit on different lines) don't fold; the tag scanner is per-line.
- **Ruby / Bash / Shell** — keyword pairs (`do … end`, `def … end`, `if … fi`, `case … esac`, etc.) aren't modeled, so most idiomatic blocks in these languages don't fold today.
- **JSX / TSX element pairs** (`<Foo>…</Foo>`) don't fold — the bracket strategy only matches `{}` `[]`. Inline `{…}` JSX expressions and `style={{…}}` braces fold normally.
- **Bracket strategy ignores `(…)`** — argument lists and tuple literals never fold. Mirrors VS Code's default.
- **Indentation strategy assumes consistent leading whitespace** within a block. Mixing tabs and spaces inside the same block can produce wrong spans.
- **Single-line constructs never fold** — a multi-line opener whose closer is on the same line emits no range, and inline objects/arrays in a multi-line parent don't get their own toggle.
